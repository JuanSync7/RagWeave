# Docling-Native Chunking Pipeline — Specification Summary

---

## 1) Generic System Overview

### Purpose

When a document parser produces a rich, structured document representation, converting that representation to a plain string and then heuristically reconstructing structure from the string is wasteful and lossy. This system redesigns the ingestion chunking pipeline to carry the structured document object all the way from parsing to chunking, using it directly to produce structure-aware, token-aligned chunks. Without this redesign, structural information — paragraph boundaries, table layouts, code block delineation, hierarchical headings — is discarded at the earliest stage and then approximated (poorly) by downstream heuristics. The result is low-quality chunks: tables split mid-row, code blocks broken mid-statement, oversized chunks silently truncated by the embedding model, and expensive LLM rewriting nodes added to compensate for what structure-aware chunking would have provided automatically.

### How It Works

The pipeline begins when a source document is parsed by the structured parser, which produces two artifacts: a plain-text markdown export and a native structured document object encoding every item (paragraph, table row, list item, figure, code block) with its type and hierarchical position. Both artifacts are threaded through Phase 1 (document processing) and persisted at the Phase 1/Phase 2 boundary — the structured object is serialized using the parser's own serialization API and written atomically alongside the existing markdown files.

At the start of Phase 2 (embedding), the orchestrator reads the persisted files. If the serialized document object is present, it is deserialized and placed in pipeline state. The path then forks: documents with a structured object route to the structure-aware chunker, which respects document item boundaries, repeats table header rows in every table chunk, merges undersized adjacent items up to the token limit, and splits oversized items at logical boundaries (e.g., line boundaries for code blocks). Documents without a structured object route to the existing plain-text chunking path, which is preserved unchanged as the fallback.

For Docling-parsed documents, the text-cleaning and paragraph-rewriting nodes are skipped because the parser already produces clean, well-formed output and structure-aware chunking eliminates the need for LLM-based paragraph rewriting. Both paths apply lightweight unicode normalization per chunk before producing the final output.

After chunking, a conditional image enrichment step replaces figure placeholders in chunk text with semantic descriptions generated by a vision model. This step runs post-chunking so that descriptions are injected into the specific chunks that contain the relevant placeholders rather than into the full document text before splitting.

### Tunable Knobs

Operators can configure the maximum chunk size in tokens. This controls how the chunker balances chunk density against the embedding model's capacity; the default is aligned to the system's standard embedding model. A storage-versus-compute toggle controls whether the serialized structured document object is persisted between phases or discarded after Phase 1 (requiring re-parsing in Phase 2 if the native path is needed). The image enrichment mode selects between a locally-run lightweight vision model, a frontier vision model accessed via a routing layer, or disabled — controlling the quality and cost of image description generation. A per-document figure count limit caps the number of images sent to the external vision model to control API cost for image-heavy documents.

### Design Rationale

The core principle is to use native structural representations when they are available rather than converting to strings and heuristically recovering structure. This eliminates an entire class of compensating nodes (text cleaning, paragraph rewriting) for documents that the structured parser handles. The pipeline uses configuration-driven path selection: the chunking path is chosen automatically based on whether the structured document object is present in state, not by a separate configuration flag, which prevents configuration conflicts. A fail-safe approach is applied throughout: if the structure-aware chunker fails for a document, the system falls back to the plain-text path using the markdown already in state; if a vision model call fails for a figure, the placeholder is left in place and the document continues processing. The decision to move image enrichment to post-chunking reflects the insight that pre-chunking image injection pollutes the full document text before splitting, causing descriptions to be split across chunk boundaries.

### Boundary Semantics

**Entry point:** A source document file parsed by the structured parser, yielding a structured document object and a markdown string. Both enter Phase 1 pipeline state at the structure detection node.

**Phase 1 exit / Phase 2 entry:** The Phase 1/Phase 2 boundary store holds markdown text, metadata, and (when applicable) the serialized structured document object. Phase 2 initialization reads all three and populates state before any Phase 2 node runs.

**Exit point:** A list of chunk objects with a standardized metadata schema — source identifiers, hierarchical section path, heading, heading level, chunk index, and total chunk count — ready for downstream enrichment and embedding. The chunking path (structured or fallback) is invisible to all downstream consumers; both paths produce the same contract.

## 3) Scope and Boundaries

**Entry point:** Source document file parsed by the structured document parser, producing a structured document object and a markdown export. Both are stored in Phase 1 pipeline state.

**Exit point:** A list of `ProcessedChunk` objects with structure-aware metadata, ready for embedding and storage.

**In scope:**
- Preservation and propagation of the structured document object through Phase 1 pipeline state
- Atomic serialization and persistence of the structured document object at the Phase 1/Phase 2 boundary store
- Structure-aware, token-aligned chunking via the native chunker for documents with a structured object
- Skipping text-cleaning and paragraph-rewriting nodes for structured-parser documents
- Automatic path selection (structured path vs. fallback) based on object presence in state
- Fallback to the existing plain-text chunking path for documents without a structured object
- Post-chunking image enrichment with three selectable modes (local model, external model, disabled)
- Per-chunk unicode normalization applied uniformly to both paths
- New configuration fields and environment variable bindings for all new tunable dimensions
- State contract changes for both Phase 1 and Phase 2 pipeline state
- Error taxonomy and fallback matrix covering chunker failures, serialization failures, VLM failures, and format-routing failures
- Supported format list and routing behavior for unsupported formats

**Out of scope (this spec):**
- Changes to the structured parser's configuration or model selection
- Embedding model changes or embedding storage schema changes
- Vector store collection schema modifications
- Knowledge graph extraction changes
- Quality validation node changes
- Chunk enrichment node changes (beyond receiving structured chunker output)
- CLI or web UI changes beyond new configuration flags
- Removal of the existing markdown pipeline code (retained as fallback)
- Re-ingestion logic changes

**Out of scope (this project):**
- Replacing the structured parser with an alternative document parser
- Multi-document cross-reference chunking
- Real-time streaming chunking
- GPU-accelerated chunking

## 4) Architecture / Pipeline Overview

```
Source Document File
    |
    v
[STRUCTURE DETECTION]
    Parse -> structured object + markdown export
    Regex fallback if parser disabled or fails
    |
    +------ Docling path? ------+
    |  Yes                      |  No
    v                           v
[SKIP text_cleaning]       [TEXT CLEANING]
[SKIP doc_refactoring]     [DOC REFACTORING]  (existing, unchanged)
    |                           |
    +---------------------------+
    v
[CLEAN DOCUMENT STORE]
    markdown + metadata + (optional) structured object JSON
    |
    +------ Structured object available? ------+
    |  Yes                                     |  No
    v                                          v
[HYBRID CHUNKER]                        [MARKDOWN CHUNKING]
  structure-aware, token-aware           MarkdownHeaderSplit ->
  section_path metadata                  RecursiveCharSplit
  table headers repeated per chunk       (existing, unchanged)
    |                                          |
    +------------------------------------------+
    v
[VLM ENRICHMENT]  (conditional: figures in chunks AND VLM enabled)
    Replace image placeholders with vision model descriptions
    Modes: local model | external model | disabled
    |
    v
[CHUNK OUTPUT]
    ProcessedChunk list
    source, section_path, chunk_index, total_chunks, ...
```

**Optional stages:** Text cleaning and document refactoring run only for the fallback (non-structured) path. VLM enrichment runs only when figures are present and VLM is not disabled.

## 7) Non-Functional and Security Themes

**Performance:**
- Structure-aware chunker execution targets sub-second completion for typical document sizes
- Serialization and deserialization round-trips are bounded to avoid becoming pipeline bottlenecks
- VLM enrichment latency targets are defined per image for both local and external modes
- Per-chunk unicode normalization is sub-millisecond

**Backward Compatibility:**
- All existing configuration fields, environment variables, and default values remain unchanged
- New fields default to values that preserve pre-redesign behavior
- Downstream nodes require zero code changes to process structure-aware chunks

**Configurability:**
- All new parameters are externalized to environment variables with documented defaults
- No new parameter is hardcoded; changes take effect on process restart

**Observability:**
- Processing log entries indicate which chunking path was used and whether VLM enrichment ran
- Errors are classified by category (transient, permanent, partial, configuration)

**Maintainability:**
- Serialized structured document format should include a version marker to support future schema migrations

## 8) Design Principles

| Principle | Description |
|-----------|-------------|
| **Structure preservation over string heuristics** | Use the native structured representation directly when available; do not convert to string and reconstruct structure. |
| **Fail-safe over fail-fast** | A single document's chunker failure or image enrichment failure must not halt a batch job; fall back gracefully. |
| **Token-alignment over character-alignment** | Chunk size limits are expressed in tokens matching the embedding model's tokenizer, not in characters. |
| **Configuration-driven path selection** | Chunking path and VLM mode are controlled by configuration and data availability, not scattered code-level conditionals. |
| **Deterministic over LLM-dependent** | Where structure-aware chunking eliminates the need for LLM-based rewriting, the deterministic approach is preferred. |

---

## 9) Key Decisions

- **Post-chunking VLM placement:** Image enrichment runs after chunking so that descriptions are injected into the specific chunks containing the relevant figure placeholders, not into the full document text before splitting.
- **Automatic path selection:** The chunking path is selected based on whether the structured document object is present in state — no separate configuration flag. This prevents misconfiguration where the parser is enabled but the chunker is not, or vice versa.
- **Unified chunk output contract:** Both the structured path and the fallback path produce identical `ProcessedChunk` metadata schemas. The chunking path is invisible to downstream consumers.
- **Node skipping for structured documents:** Text-cleaning and paragraph-rewriting nodes are skipped for structured-parser documents because the parser produces clean output and structure-aware chunking renders LLM rewriting redundant.
- **Atomic serialization at the phase boundary:** The structured document object is written using a tmp-file-then-rename pattern to prevent partial writes from producing corrupt files that crash Phase 2.
- **Storage-vs-compute toggle:** Operators who are storage-constrained can disable structured object persistence; Phase 2 then re-parses the source document or falls back to markdown chunking.
- **Format-error vs. parser-error separation:** An unsupported file format always routes to the fallback path regardless of strict-mode setting; strict mode applies only to genuine parser failures on supported formats.

## 10) Acceptance and Evaluation

The spec defines a system-level acceptance criteria table covering seven checkpoints:

- 100% of structured-parser documents use the structure-aware chunker when the structured object is available
- Fallback path output is byte-identical to pre-redesign output (unicode normalization is the sole deliberate delta)
- Zero code changes required in downstream pipeline nodes
- 100% of multi-chunk tables include the header row in every chunk
- Zero document-level failures caused by VLM errors
- Zero existing environment variables renamed or removed
- Zero chunks exceed the configured token limit (structure-aware path)

The spec includes a requirements traceability matrix mapping all 45 requirements to sections and components.

## 11) External Dependencies

**Required:**
- Structured document parser library and its core data model package (for structured object type, serialization API, and structure-aware chunker)
- Embedding model with a known maximum token input length (drives the default max-tokens configuration)

**Optional (mode-dependent):**
- Locally-run lightweight vision model (required when VLM mode is set to local)
- LiteLLM routing layer (required when VLM mode is set to external)

**Downstream contract:**
- `ProcessedChunk` object (text + metadata dictionary) — the stable output contract consumed by chunk enrichment, metadata generation, cross-reference extraction, and embedding storage nodes
