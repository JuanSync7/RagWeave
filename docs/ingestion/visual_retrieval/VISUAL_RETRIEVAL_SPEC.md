## 1. Scope & Definitions

### 1.1 Problem Statement

The RAG system currently retrieves information exclusively through text-based chunking and embedding (BGE-M3 hybrid search). Documents with rich visual content -- charts, diagrams, tables rendered as images, presentation slides, and layout-dependent pages -- lose semantic information when reduced to extracted text. A ColQwen2-based visual embedding pipeline has been built on the ingestion side: page images are extracted via Docling, embedded through ColQwen2 into 128-dimensional vectors, stored in MinIO, and indexed in a dedicated Weaviate collection (`RAGVisualPages`). However, no retrieval path exists to query this visual index. Users cannot find visually-relevant document pages because the system has no mechanism to encode a text query into the ColQwen2 embedding space and search the visual collection.

### 1.2 Scope

This specification defines the requirements for the **visual page retrieval pipeline** of the RAG system. The boundary is:

- **Entry point:** A text query arrives at the RAG pipeline with visual retrieval enabled via configuration.
- **Exit point:** The pipeline response includes a list of visually-matched document pages, each with a similarity score and a time-limited presigned URL for the page image.

Everything between these two points is in scope. The ingestion side (page image extraction, visual embedding, storage, and indexing) is already implemented and out of scope.

### 1.3 Terminology

| Term | Definition |
|------|-----------|
| **ColQwen2** | A vision-language model (`vidore/colqwen2-v1.0`) that produces 128-dimensional patch-level embeddings from document page images, and can encode text queries into the same 128-dimensional space for cross-modal retrieval. |
| **Mean vector** | A single 128-dimensional vector computed as the arithmetic mean of all patch vectors for a page image. Used as the primary retrieval vector in Weaviate. |
| **Patch vector** | One 128-dimensional vector per spatial patch of a page image, produced by ColQwen2. Stored but not used for retrieval in this spec. |
| **Visual collection** | The `RAGVisualPages` Weaviate collection that stores per-page visual embeddings with metadata and a named vector (`mean_vector`). |
| **Text track** | The existing retrieval path: BGE-M3 embedding, Weaviate hybrid search, BGE reranking, LLM generation. |
| **Visual track** | The new retrieval path specified here: ColQwen2 text encoding, Weaviate near-vector search on the visual collection, presigned URL generation. |
| **Presigned URL** | A time-limited URL generated by MinIO that grants temporary read access to a specific object without requiring authentication credentials. |
| **RAGChain** | The orchestrator class in `src/retrieval/pipeline/rag_chain.py` that executes the full retrieval pipeline. |

### 1.4 Requirement Priority Levels

This document uses RFC 2119 language:

- **MUST** -- Absolute requirement. The system is non-conformant without it.
- **SHOULD** -- Recommended. May be omitted only with documented justification.
- **MAY** -- Optional. Included at the implementor's discretion.

### 1.5 Requirement Format

Each requirement follows this structure:

> **FR-xxx** | Priority: MUST/SHOULD/MAY
> **Description:** What the system shall do.
> **Rationale:** Why this requirement exists.
> **Acceptance Criteria:** How to verify conformance.

This document uses domain-prefixed IDs:

- **FR-xxx** -- Functional Requirements
- **NFR-xxx** -- Non-Functional Requirements

Requirements are grouped by section with the following ID ranges:

| Section | ID Range | Component |
|---------|----------|-----------|
| Section 3 | FR-1xx | Configuration |
| Section 4 | FR-2xx | ColQwen2 Text Query Encoding |
| Section 5 | FR-3xx | Weaviate Visual Search |
| Section 6 | FR-4xx | MinIO Presigned URL Generation |
| Section 7 | FR-5xx | Retrieval Schema Extension |
| Section 8 | FR-6xx | RAGChain Visual Track Integration |
| Section 9 | FR-7xx | Server API Schema Extension |
| Section 10 | NFR-9xx | Non-Functional Requirements |

### 1.6 Assumptions & Constraints

| ID | Assumption / Constraint | Impact if Violated |
|----|------------------------|--------------------|
| A-1 | Python 3.11+ runtime is available | Type hint syntax (e.g., `list[float]`) will fail on older versions |
| A-2 | ColQwen2 model (`vidore/colqwen2-v1.0`) is accessible from Hugging Face Hub or local cache | Model load will fail; visual retrieval will be unavailable |
| A-3 | CUDA-capable GPU with at least 6 GB VRAM is present (RTX 2060 or equivalent) | ColQwen2 text encoder cannot load alongside BGE-M3 and reranker |
| A-4 | The `RAGVisualPages` Weaviate collection exists and contains indexed visual embeddings from the ingestion pipeline | Visual search will return zero results |
| A-5 | MinIO is running and the configured bucket contains page images at the expected key pattern (`pages/{document_id}/{page_number:04d}.jpg`) | Presigned URLs will resolve to 404 errors |
| A-6 | The `colpali-engine` and `bitsandbytes` packages are installed | ColQwen2 model load will raise `ColQwen2LoadError` |
| A-7 | ColQwen2 uses 4-bit BitsAndBytes quantization; text encoder at 4-bit occupies approximately 1-1.5 GB VRAM | VRAM budget calculations are based on this estimate |

### 1.7 Design Principles

| Principle | Description |
|-----------|-------------|
| **Additive integration** | The visual track is added alongside the existing text track without modifying text-track behavior. When visual retrieval is disabled, the system behaves identically to the pre-visual baseline. |
| **Lazy resource allocation** | GPU-heavy resources (ColQwen2 model) are loaded only when first needed, not at system startup. This avoids VRAM consumption when visual retrieval is enabled but no visual queries have been issued. |
| **Sequential GPU access** | Visual retrieval stages execute after text retrieval stages complete, never concurrently, to prevent VRAM contention on constrained hardware. |
| **Configuration-driven behavior** | All visual retrieval behavior is controlled by typed configuration keys with sensible defaults. No visual retrieval behavior is hardcoded. |
| **Single-instance infrastructure** | The visual track reuses existing Weaviate and MinIO instances. No new infrastructure services are introduced. |

### 1.8 Out of Scope

**Out of scope -- this spec (already implemented in companion ingestion specs):**

- Docling page image extraction from PDF, PPTX, DOCX
- ColQwen2 image embedding (`embed_page_images()`)
- MinIO page image storage (`store_page_images()`, `delete_page_images()`)
- Weaviate visual collection schema creation (`ensure_visual_collection()`)
- Weaviate visual document insertion (`add_visual_documents()`)
- Weaviate visual document deletion (`delete_visual_by_source_key()`)
- Configuration keys for visual embedding during ingestion (`RAG_INGESTION_ENABLE_VISUAL_EMBEDDING`, `RAG_INGESTION_COLQWEN_MODEL`, etc.)
- Standalone image file handling during ingestion

**Out of scope -- this project (not planned for any phase):**

- Image-to-image queries (user uploads an image to find similar pages)
- Visual reranking (cross-encoder reranking of visual results)
- Patch-level MaxSim retrieval (using individual patch vectors instead of mean vector)
- Multi-vector late interaction scoring (ColBERT-style)
- Visual result summarization (generating text descriptions of retrieved page images at query time)
- UI rendering of visual results (frontend concern)
- Multi-instance Weaviate or MinIO deployment for visual workloads
- ColSmol or alternative visual embedding model support
- OCR fallback for pages without text extraction

### 2.2 Data Flow Summary

| Stage | Input | Output |
|-------|-------|--------|
| ColQwen2 Text Encoding | Processed query text (string) | 128-dim float vector |
| Weaviate Visual Search | 128-dim query vector, limit, score threshold | List of page objects with metadata and similarity scores |
| Presigned URL Generation | MinIO key for each matched page | Time-limited presigned URL per page |
| Response Assembly | Text results (existing) + visual page results | Unified RAGResponse with both result types |

## 3. Configuration

> **FR-101** | Priority: MUST
>
> **Description:** The system MUST provide a boolean configuration key `RAG_VISUAL_RETRIEVAL_ENABLED` that controls whether the visual retrieval track is active. The default value MUST be `false`.
>
> **Rationale:** Visual retrieval requires GPU resources (ColQwen2 model) and an indexed visual collection. Operators must explicitly opt in after confirming that the ingestion pipeline has populated the visual collection and that sufficient VRAM is available. Supports the configuration-driven behavior principle.
>
> **Acceptance Criteria:** When `RAG_VISUAL_RETRIEVAL_ENABLED=false` (or unset), the RAGChain produces a response with `visual_results` set to `None` regardless of whether the visual collection contains data. When `RAG_VISUAL_RETRIEVAL_ENABLED=true`, the visual track executes as specified.

> **FR-103** | Priority: MUST
>
> **Description:** The system MUST provide an integer configuration key `RAG_VISUAL_RETRIEVAL_LIMIT` that controls the maximum number of visual page results returned per query. The default value MUST be `5`. Valid range: 1 to 50.
>
> **Rationale:** Returning too many visual results increases response payload size and presigned URL generation time without proportional value to the user. A configurable limit lets operators balance result coverage against response latency.
>
> **Acceptance Criteria:** When `RAG_VISUAL_RETRIEVAL_LIMIT=3`, at most 3 visual page results are returned. When set to a value outside the valid range (0 or 51), the system logs a warning and clamps to the nearest boundary.

> **FR-105** | Priority: MUST
>
> **Description:** The system MUST provide a float configuration key `RAG_VISUAL_RETRIEVAL_MIN_SCORE` that sets the minimum cosine similarity threshold for visual results. Pages with similarity below this threshold MUST be excluded from results. The default value MUST be `0.3`. Valid range: 0.0 to 1.0.
>
> **Rationale:** Low-similarity visual results are noise. A configurable threshold allows operators to tune precision versus recall based on their corpus characteristics.
>
> **Acceptance Criteria:** When `RAG_VISUAL_RETRIEVAL_MIN_SCORE=0.5`, pages with cosine similarity below 0.5 are excluded from the visual results list. When all candidate pages score below the threshold, the visual results list is empty (not `None`).

> **FR-107** | Priority: SHOULD
>
> **Description:** The system SHOULD provide an integer configuration key `RAG_VISUAL_RETRIEVAL_URL_EXPIRY_SECONDS` that controls the expiry duration of presigned MinIO URLs for page images. The default value SHOULD be `3600` (1 hour). Valid range: 60 to 86400.
>
> **Rationale:** Presigned URLs should expire within a bounded time window to limit exposure of page images. The default of 1 hour balances usability (URLs remain valid during a typical user session) against security (URLs do not persist indefinitely).
>
> **Acceptance Criteria:** When `RAG_VISUAL_RETRIEVAL_URL_EXPIRY_SECONDS=1800`, generated presigned URLs expire after 30 minutes. When unset, URLs expire after 3600 seconds.

> **FR-109** | Priority: MUST
>
> **Description:** The system MUST reuse the existing `RAG_INGESTION_COLQWEN_MODEL` configuration key to determine which ColQwen2 model to load for text query encoding at retrieval time.
>
> **Rationale:** The query encoder and the image encoder must use the same model to produce embeddings in the same vector space. Introducing a separate retrieval-time model key would allow misconfiguration where the query encoder and image encoder use different models, producing incompatible embeddings. A single source of truth for the model name eliminates this class of error.
>
> **Acceptance Criteria:** When `RAG_INGESTION_COLQWEN_MODEL=vidore/colqwen2-v1.0`, the retrieval pipeline loads the same model for text query encoding. No separate `RAG_VISUAL_RETRIEVAL_MODEL` key exists.

> **FR-111** | Priority: MUST
>
> **Description:** The system MUST validate all visual retrieval configuration keys at startup and fail fast with a clear error message if contradictory or invalid settings are detected.
>
> **Rationale:** Catching configuration errors early prevents silent failures at query time (e.g., visual retrieval enabled but the visual collection name is empty, or the score threshold exceeds 1.0). Supports the configuration-driven behavior principle.
>
> **Acceptance Criteria:** When `RAG_VISUAL_RETRIEVAL_ENABLED=true` and `RAG_INGESTION_VISUAL_TARGET_COLLECTION` is empty, the system raises a configuration error at startup with a message identifying both keys and the conflict. When `RAG_VISUAL_RETRIEVAL_MIN_SCORE=1.5`, the system raises a configuration error identifying the out-of-range value.

## 4. ColQwen2 Text Query Encoding

> **FR-201** | Priority: MUST
>
> **Description:** The system MUST provide a function `embed_text_query(model, processor, text)` in the ColQwen2 adapter module that encodes a text query into a 128-dimensional float vector using ColQwen2's text encoding pathway.
>
> **Rationale:** ColQwen2's image encoder produces 128-dim vectors during ingestion. To search the same embedding space, text queries must be encoded through ColQwen2's text encoder (via `processor.process_queries()`), not through BGE-M3 or any other model. Without a text encoding function, there is no way to produce query vectors compatible with the visual collection's indexed vectors.
>
> **Acceptance Criteria:** Given a non-empty text query, `embed_text_query()` returns a `list[float]` of exactly 128 elements. Given the same query text and model state, repeated calls produce identical vectors (deterministic under `torch.inference_mode()`).

> **FR-203** | Priority: MUST
>
> **Description:** The `embed_text_query` function MUST use `processor.process_queries()` to tokenize the input text, pass the tokenized input through the model in inference mode, and produce a single 128-dimensional vector by mean-pooling across token-level output vectors.
>
> **Rationale:** ColQwen2's architecture produces per-token patch vectors for text queries (analogous to per-patch vectors for images). The mean-pooling strategy matches the mean-pooling used during ingestion for page images, ensuring that query and document vectors are in the same representational space. Using a different pooling strategy would create a distribution mismatch.
>
> **Acceptance Criteria:** The output vector is the arithmetic mean of all token-level vectors from the model's forward pass. The computation uses `torch.inference_mode()` to prevent gradient computation. The vector dtype is float32 (Python `float`), not float16 or bfloat16.

> **FR-205** | Priority: MUST
>
> **Description:** The `embed_text_query` function MUST handle empty or whitespace-only input text by raising a `ValueError` with a descriptive message, without invoking the model.
>
> **Rationale:** Passing empty input to ColQwen2's processor may produce undefined behavior or a zero vector. Failing fast with a clear error is safer and more debuggable than silently returning a degenerate embedding.
>
> **Acceptance Criteria:** Given an empty string or a string containing only whitespace, `embed_text_query()` raises `ValueError` with a message containing "empty" or "blank". The model is not invoked.

> **FR-207** | Priority: MUST
>
> **Description:** The `embed_text_query` function MUST propagate `ColQwen2LoadError` if the model or processor is `None` or invalid, and MUST wrap unexpected inference errors in `VisualEmbeddingError` with the original exception chained.
>
> **Rationale:** Callers (RAGChain) need typed exceptions to distinguish between fatal model errors (do not retry, disable visual track) and transient inference errors (log and return empty visual results). Using the existing exception hierarchy from the ColQwen2 adapter module maintains consistency with ingestion-side error handling.
>
> **Acceptance Criteria:** When the model reference is `None`, `ColQwen2LoadError` is raised. When the model's forward pass raises a `RuntimeError` (e.g., CUDA OOM), `VisualEmbeddingError` is raised with the `RuntimeError` as the `__cause__`.

## 5. Weaviate Visual Search

> **FR-301** | Priority: MUST
>
> **Description:** The system MUST provide a function `visual_search()` in the Weaviate visual store module that performs nearest-neighbor search on the `mean_vector` named vector of the `RAGVisualPages` collection, returning page objects ranked by cosine similarity.
>
> **Rationale:** The visual collection has indexed page embeddings since the ingestion pipeline was built, but there is no function to query them. This function completes the retrieval path by enabling vector-based search on the visual collection using the same cosine distance metric used during indexing.
>
> **Acceptance Criteria:** Given a 128-dim query vector, `visual_search()` returns a list of dicts, each containing: `document_id` (str), `page_number` (int), `source_key` (str), `source_name` (str), `minio_key` (str), `tenant_id` (str), `total_pages` (int), `page_width_px` (int), `page_height_px` (int), and `score` (float, cosine similarity). Results are ordered by descending score.

> **FR-303** | Priority: MUST
>
> **Description:** The `visual_search()` function MUST accept a `limit` parameter controlling the maximum number of results and a `score_threshold` parameter controlling the minimum cosine similarity. Pages with similarity below `score_threshold` MUST be excluded from results.
>
> **Rationale:** Without server-side filtering by score threshold, the caller must retrieve all results and filter client-side, which wastes network bandwidth and adds latency. Parameterizing both limit and threshold allows the pipeline to delegate filtering to Weaviate's query engine, which is more efficient.
>
> **Acceptance Criteria:** When `limit=5` and `score_threshold=0.4`, at most 5 results are returned, all with cosine similarity >= 0.4. When no pages meet the threshold, an empty list is returned (not `None`).

> **FR-305** | Priority: MUST
>
> **Description:** The `visual_search()` function MUST accept an optional `tenant_id` parameter. When provided, results MUST be filtered to only include pages whose `tenant_id` property matches the provided value.
>
> **Rationale:** Multi-tenant deployments require that visual search results are scoped to the requesting tenant. Without tenant filtering, visual search would leak page images across tenant boundaries. This matches the existing text track's tenant filtering behavior.
>
> **Acceptance Criteria:** When `tenant_id="tenant_a"`, only pages with `tenant_id="tenant_a"` are returned. When `tenant_id=None`, no tenant filter is applied (all pages are candidates).

> **FR-307** | Priority: MUST
>
> **Description:** The `visual_search()` function MUST accept an optional `collection` parameter that defaults to the value of `RAG_INGESTION_VISUAL_TARGET_COLLECTION` (default: `"RAGVisualPages"`).
>
> **Rationale:** The collection name is parameterized in all existing visual store functions (`ensure_visual_collection()`, `add_visual_documents()`, `delete_visual_by_source_key()`). The search function must follow the same pattern for consistency and to support custom collection names.
>
> **Acceptance Criteria:** When `collection="CustomVisualPages"`, the search queries the `CustomVisualPages` collection. When `collection=None` or omitted, the default collection is used.

> **FR-309** | Priority: MUST
>
> **Description:** The `visual_search()` function MUST use the `near_vector` query method on the `mean_vector` named vector with cosine distance metric for similarity computation.
>
> **Rationale:** The `RAGVisualPages` collection is configured with HNSW indexing and cosine distance on the `mean_vector` named vector. Using a different query method or distance metric would produce incorrect or inconsistent similarity scores relative to the indexed vectors.
>
> **Acceptance Criteria:** The Weaviate query uses `near_vector` targeting the named vector `mean_vector`. Returned scores represent cosine similarity (not cosine distance). A perfect match returns a score of 1.0.

> **FR-311** | Priority: MUST
>
> **Description:** The `visual_search()` function MUST NOT return the `patch_vectors` property in search results.
>
> **Rationale:** Patch vectors are stored as JSON-serialized text and can be hundreds of kilobytes per page. Including them in search results would dramatically increase response payload size and memory usage without benefit, since patch-level retrieval is out of scope. The existing `add_visual_documents()` function stores patch vectors for future MaxSim use, but retrieval does not need them.
>
> **Acceptance Criteria:** The returned dicts do not contain a `patch_vectors` key. The Weaviate query explicitly excludes the `patch_vectors` property from the response.

> **FR-313** | Priority: MUST
>
> **Description:** The `search_visual()` method MUST be added to the `VectorBackend` abstract base class as an abstract method, and implemented in the Weaviate backend. The public API module MUST export a `search_visual()` function that delegates to the backend.
>
> **Rationale:** All vector store operations follow the backend abstraction pattern: abstract method in `VectorBackend`, concrete implementation in the Weaviate backend, and a public function in `vector_db/__init__.py`. Adding visual search without extending this contract would create an inconsistency where visual search bypasses the backend abstraction.
>
> **Acceptance Criteria:** `VectorBackend` has an abstract `search_visual()` method. The Weaviate backend implements it by delegating to `visual_search()`. `src/vector_db/__init__.py` exports `search_visual()`. Calling `search_visual()` from the public API produces the same results as calling `visual_search()` directly on the store module.

## 6. MinIO Presigned URL Generation

> **FR-401** | Priority: MUST
>
> **Description:** The system MUST provide a function `get_page_image_url(client, minio_key, bucket, expires_in_seconds)` in the MinIO store module that generates a presigned GET URL for an arbitrary object key.
>
> **Rationale:** The existing `get_document_url()` function appends a `.md` content suffix to the document ID, making it unsuitable for page image keys (which follow the pattern `pages/{document_id}/{page_number:04d}.jpg`). A new function that accepts a raw MinIO key is needed to generate presigned URLs for page images without key transformation.
>
> **Acceptance Criteria:** Given `minio_key="pages/abc123/0001.jpg"` and `expires_in_seconds=3600`, the function returns a presigned URL that grants GET access to that object for 3600 seconds. The function does not append any suffix to the key. If the object does not exist, the function still returns a presigned URL (MinIO generates presigned URLs without verifying object existence; the URL will return 404 when accessed).

> **FR-403** | Priority: MUST
>
> **Description:** The `get_page_image_url()` function MUST default the `bucket` parameter to the configured `MINIO_BUCKET` and the `expires_in_seconds` parameter to the value of `RAG_VISUAL_RETRIEVAL_URL_EXPIRY_SECONDS`.
>
> **Rationale:** Callers should not need to specify the bucket or expiry on every call when the defaults are correct. Using the configured expiry ensures consistency with the operator's security policy.
>
> **Acceptance Criteria:** When called with only `client` and `minio_key`, the function uses the configured bucket and expiry. When called with explicit values, those values override the defaults.

## 7. Retrieval Schema Extension

> **FR-501** | Priority: MUST
>
> **Description:** The system MUST define a `VisualPageResult` dataclass in the retrieval schemas module with the following fields: `document_id` (str), `page_number` (int), `source_key` (str), `source_name` (str), `score` (float), `page_image_url` (str), `total_pages` (int), `page_width_px` (int), `page_height_px` (int).
>
> **Rationale:** Visual retrieval results have a fundamentally different structure from text chunk results (`RankedResult`). A text result carries a `text` field and metadata dict; a visual result carries a page image URL, page dimensions, and page-level metadata. A dedicated dataclass provides type safety and makes the visual result contract explicit.
>
> **Acceptance Criteria:** `VisualPageResult` is importable from `src.retrieval.common.schemas`. All fields are present with the specified types. The dataclass can be instantiated with all fields. `score` represents cosine similarity (0.0 to 1.0). `page_image_url` is a string containing a presigned MinIO URL.

> **FR-503** | Priority: MUST
>
> **Description:** The `RAGResponse` dataclass MUST include an optional field `visual_results` of type `Optional[List[VisualPageResult]]`, defaulting to `None`.
>
> **Rationale:** The visual results field must be optional to maintain backward compatibility with existing text-only responses. When visual retrieval is disabled, `visual_results` remains `None`, and no serialization or deserialization cost is incurred. This follows the additive integration principle.
>
> **Acceptance Criteria:** An existing `RAGResponse` constructed without `visual_results` has `visual_results=None`. A response with visual retrieval enabled has `visual_results` as a list of `VisualPageResult` instances (possibly empty if no pages matched the threshold). Existing code that reads `RAGResponse` fields other than `visual_results` is unaffected.

## 8. RAGChain Visual Track Integration

> **FR-601** | Priority: MUST
>
> **Description:** When `RAG_VISUAL_RETRIEVAL_ENABLED=true`, the RAGChain MUST execute the visual retrieval track after all text track stages (query processing, KG expansion, embedding, hybrid search, reranking) have completed and before the final response is returned.
>
> **Rationale:** Sequential execution prevents VRAM contention between the text track's GPU models (BGE-M3, BGE-reranker) and the visual track's GPU model (ColQwen2). Running them concurrently risks CUDA out-of-memory errors on the RTX 2060. Supports the sequential GPU access principle.
>
> **Acceptance Criteria:** When visual retrieval is enabled, the visual search stage timing entry appears after the reranking stage timing entry in the response's `stage_timings` list. No visual retrieval function is invoked before text reranking completes.

> **FR-603** | Priority: MUST
>
> **Description:** The RAGChain MUST load the ColQwen2 model lazily on the first query that triggers visual retrieval, not during `__init__()`. The loaded model and processor MUST be cached as instance attributes for subsequent queries.
>
> **Rationale:** Loading ColQwen2 at startup consumes VRAM even if no visual queries are issued. Lazy loading defers VRAM allocation until the resource is actually needed, which is critical when the system may run for extended periods without visual queries. Supports the lazy resource allocation principle.
>
> **Acceptance Criteria:** After `RAGChain.__init__()` completes with `RAG_VISUAL_RETRIEVAL_ENABLED=true`, no ColQwen2 model is loaded (VRAM usage is unchanged from the text-only baseline). After the first query that triggers visual retrieval, the ColQwen2 model is loaded and cached. The second visual query reuses the cached model without reloading.

> **FR-605** | Priority: MUST
>
> **Description:** The RAGChain visual track MUST use the processed query text (the output of the query processing stage) as the input to `embed_text_query()`, not the raw user query.
>
> **Rationale:** The query processing stage reformulates the raw query for better retrieval (expanding abbreviations, resolving coreferences, etc.). Using the processed query for visual embedding produces better-aligned vectors than using the raw query, since the processed query is also what drives the text track's hybrid search.
>
> **Acceptance Criteria:** When the query processing stage reformulates "what does fig 3 show?" to "what does figure 3 illustrate in the document?", the visual track embeds "what does figure 3 illustrate in the document?", not "what does fig 3 show?".

> **FR-607** | Priority: MUST
>
> **Description:** After retrieving visual search results from Weaviate, the RAGChain MUST generate a presigned MinIO URL for each result's `minio_key` and attach the URL to the corresponding `VisualPageResult`'s `page_image_url` field.
>
> **Rationale:** Weaviate stores the MinIO object key, not a URL. The presigned URL is needed for callers (API clients, UI) to directly access the page image without additional MinIO credentials. Generating the URL at retrieval time ensures that the URL is fresh and not expired.
>
> **Acceptance Criteria:** Each `VisualPageResult` in the response has a non-empty `page_image_url` that is a valid presigned MinIO URL. The URL points to the correct page image key. The URL's expiry matches the configured `RAG_VISUAL_RETRIEVAL_URL_EXPIRY_SECONDS`.

> **FR-609** | Priority: MUST
>
> **Description:** The RAGChain MUST pass the request's `tenant_id` to the visual search function, ensuring visual results are scoped to the same tenant as text results.
>
> **Rationale:** A query from tenant A must not receive visual pages belonging to tenant B. The text track already filters by tenant; the visual track must apply the same filter to maintain tenant isolation.
>
> **Acceptance Criteria:** When a query arrives with `tenant_id="tenant_a"`, visual search results only include pages with `tenant_id="tenant_a"`.

> **FR-611** | Priority: MUST
>
> **Description:** The RAGChain MUST record the visual retrieval stage timing (wall-clock duration in milliseconds) in the response's `stage_timings` list with a stage name of `"visual_retrieval"`.
>
> **Rationale:** All existing pipeline stages record their timing in `stage_timings` for observability. The visual track must follow the same pattern to allow operators to identify visual retrieval as a latency contributor and tune configuration accordingly.
>
> **Acceptance Criteria:** When visual retrieval executes, the `stage_timings` list contains an entry with `stage="visual_retrieval"` and `duration_ms` set to the elapsed time. When visual retrieval is disabled, no `visual_retrieval` entry appears.

> **FR-613** | Priority: MUST
>
> **Description:** The RAGChain `close()` method MUST unload the ColQwen2 model (if loaded) by calling `unload_colqwen_model()` and release the cached model and processor references.
>
> **Rationale:** The `close()` method is the lifecycle cleanup point for all pipeline resources (Weaviate client, etc.). Failing to unload ColQwen2 on close would leak VRAM until process termination, preventing other GPU workloads from using the freed memory.
>
> **Acceptance Criteria:** After `RAGChain.close()`, the ColQwen2 model reference is `None`. VRAM usage returns to pre-load levels (within 200 MB tolerance, per the existing `unload_colqwen_model()` contract).

> **FR-615** | Priority: MUST
>
> **Description:** When `RAG_VISUAL_RETRIEVAL_ENABLED=false`, the RAGChain MUST NOT load the ColQwen2 model, MUST NOT execute any visual retrieval logic, and MUST set `visual_results=None` on the response.
>
> **Rationale:** When visual retrieval is disabled, the system must behave identically to the pre-visual baseline with zero additional overhead. This supports the additive integration principle: the feature is purely additive and has no cost when turned off.
>
> **Acceptance Criteria:** With `RAG_VISUAL_RETRIEVAL_ENABLED=false`, no ColQwen2-related imports are triggered at query time, no visual search is attempted, no presigned URLs are generated, and `response.visual_results is None`.

> **FR-617** | Priority: SHOULD
>
> **Description:** The RAGChain visual track SHOULD include a stage budget entry `RAG_STAGE_BUDGET_VISUAL_RETRIEVAL_MS` (default: 10000) that limits the wall-clock time for the visual retrieval stage. If the budget is exhausted, the visual track SHOULD return partial results (whatever has been retrieved so far) rather than blocking the entire response.
>
> **Rationale:** ColQwen2 model loading on the first query can take several seconds. Without a budget, a slow visual retrieval stage could cause the overall pipeline to exceed its timeout. A stage budget allows the system to gracefully degrade by returning text results with partial or empty visual results rather than timing out the entire request.
>
> **Acceptance Criteria:** When the visual retrieval stage exceeds its budget, `visual_results` is set to whatever results were retrieved before the budget expired (which may be an empty list). The response includes `budget_exhausted_stage="visual_retrieval"` when the visual stage's budget was the one exhausted.

## 9. Server API Schema Extension

> **FR-701** | Priority: MUST
>
> **Description:** The server API schemas MUST include a `VisualPageResultResponse` Pydantic model with the same fields as the `VisualPageResult` dataclass: `document_id` (str), `page_number` (int), `source_key` (str), `source_name` (str), `score` (float), `page_image_url` (str), `total_pages` (int), `page_width_px` (int), `page_height_px` (int).
>
> **Rationale:** The API response must use Pydantic models for serialization and validation, not raw dataclasses. A Pydantic model ensures that the API contract is validated, documented (via OpenAPI schema generation), and serializable to JSON.
>
> **Acceptance Criteria:** `VisualPageResultResponse` is importable from the server schemas module. It serializes to JSON with all specified fields. It is included in the auto-generated OpenAPI schema.

> **FR-703** | Priority: MUST
>
> **Description:** The `QueryResponse` Pydantic model MUST include an optional field `visual_results` of type `Optional[list[VisualPageResultResponse]]`, defaulting to `None`.
>
> **Rationale:** API clients need to receive visual results alongside text results in the query response. The field must be optional to maintain backward compatibility: existing clients that do not expect visual results will ignore the `None` value. New clients can opt into processing visual results by checking for a non-`None` value.
>
> **Acceptance Criteria:** Existing API responses serialize correctly when `visual_results` is `None` (the field is either absent or `null` in JSON). When visual results are present, the field serializes as an array of `VisualPageResultResponse` objects.

## 10. Non-Functional Requirements

> **NFR-901** | Priority: SHOULD
>
> **Description:** The visual retrieval stage (text encoding + visual search + presigned URL generation) SHOULD complete within the following performance targets on warm model (ColQwen2 already loaded):
>
> | Component | Target (P95) |
> |-----------|-------------|
> | ColQwen2 text encoding | < 200ms |
> | Weaviate visual search | < 500ms |
> | Presigned URL generation (per result) | < 50ms |
> | **Total visual stage (warm)** | **< 2000ms** |
>
> On cold start (first query, model load required), the total visual stage MAY take up to 30 seconds due to model loading.
>
> **Rationale:** The visual track adds latency on top of the existing text track. Keeping the warm-model visual stage under 2 seconds ensures that the combined pipeline (text + visual) remains within interactive response time expectations. Cold start latency is acceptable because it occurs only once per RAGChain lifecycle.
>
> **Acceptance Criteria:** Under warm-model conditions with a visual collection containing 1000 pages, the visual retrieval stage completes within 2000ms at P95 as measured by `stage_timings["visual_retrieval"]`.

> **NFR-903** | Priority: MUST
>
> **Description:** The ColQwen2 text encoder at 4-bit quantization MUST consume no more than 2 GB of VRAM when loaded alongside the existing BGE-M3 embedding model and BGE reranker. Total VRAM usage across all three models MUST remain below 5.5 GB to leave headroom on a 6 GB GPU.
>
> **Rationale:** The target deployment hardware is an NVIDIA RTX 2060 with 6 GB VRAM. BGE-M3 and BGE-reranker together consume approximately 2-3 GB. If ColQwen2's text encoder exceeds 2 GB, the three models cannot coexist, requiring a load/unload strategy that adds significant latency.
>
> **Acceptance Criteria:** With all three models loaded, `torch.cuda.memory_allocated()` reports less than 5.5 GB. If ColQwen2 text encoder isolation is not feasible (full model required), this requirement triggers the fallback strategy documented in Appendix D, Open Question 1.

> **NFR-905** | Priority: MUST
>
> **Description:** The system MUST degrade gracefully when the visual retrieval track encounters an error. Visual track failures MUST NOT prevent text track results from being returned.
>
> | Failure Mode | Degraded Behavior |
> |-------------|-------------------|
> | ColQwen2 model fails to load | `visual_results=[]`, warning logged, text results returned normally |
> | ColQwen2 text encoding fails | `visual_results=[]`, warning logged, text results returned normally |
> | Weaviate visual search fails | `visual_results=[]`, warning logged, text results returned normally |
> | MinIO presigned URL generation fails for one page | That page is omitted from `visual_results`, warning logged, remaining pages included |
> | Visual collection does not exist | `visual_results=[]`, warning logged once, text results returned normally |
>
> **Rationale:** The visual track is an enhancement to the text track, not a prerequisite. Users must always receive text results even when the visual track is completely broken. Crashing the entire pipeline because of a visual track error would be a regression in reliability. Supports the additive integration principle.
>
> **Acceptance Criteria:** For each failure mode in the table, the system returns a valid `RAGResponse` with text results intact and `visual_results` set to an empty list (not `None`, which would indicate visual retrieval is disabled, not failed). A warning-level log entry identifies the failure.

> **NFR-907** | Priority: MUST
>
> **Description:** All configurable thresholds, limits, and parameters for visual retrieval MUST be externalized to environment variable-backed configuration keys (not hardcoded in source code). Changes to configuration MUST take effect on process restart without code changes.
>
> Configuration keys covered by this requirement:
> - `RAG_VISUAL_RETRIEVAL_ENABLED` (FR-101)
> - `RAG_VISUAL_RETRIEVAL_LIMIT` (FR-103)
> - `RAG_VISUAL_RETRIEVAL_MIN_SCORE` (FR-105)
> - `RAG_VISUAL_RETRIEVAL_URL_EXPIRY_SECONDS` (FR-107)
> - `RAG_STAGE_BUDGET_VISUAL_RETRIEVAL_MS` (FR-617)
>
> **Rationale:** Hardcoded values require code changes and redeployment to tune. Environment variable-backed keys allow operators to adjust behavior without modifying source code. This is consistent with all existing configuration keys in the system.
>
> **Acceptance Criteria:** Every parameter listed above is loaded from an environment variable at module import time in `config/settings.py`. Missing environment variables fall back to the documented default values. No literal threshold values appear in retrieval pipeline code.

> **NFR-909** | Priority: MUST
>
> **Description:** The visual retrieval stage MUST emit observability spans via the existing tracer interface (`get_tracer()`) for: ColQwen2 model loading, text query encoding, Weaviate visual search, and presigned URL generation.
>
> **Rationale:** The existing text track emits spans for every stage (query processing, embedding, hybrid search, reranking, generation). Without spans for the visual track, operators cannot diagnose latency issues or monitor visual retrieval health. The observability contract must be symmetric between text and visual tracks.
>
> **Acceptance Criteria:** When visual retrieval executes, at least four spans are emitted: `visual_retrieval.model_load` (only on cold start), `visual_retrieval.text_encode`, `visual_retrieval.search`, and `visual_retrieval.presigned_urls`. Each span includes relevant attributes (e.g., query length, result count, duration).

> **NFR-911** | Priority: SHOULD
>
> **Description:** The system SHOULD log visual retrieval activity at INFO level, including: number of visual results returned, whether the model was loaded (cold start vs. warm), and the visual stage duration. Debug-level logs SHOULD include the query vector dimensions and the score range of returned results.
>
> **Rationale:** Operators need visibility into visual retrieval behavior without enabling full debug logging. INFO-level logs provide operational awareness; debug-level logs provide diagnostic detail for troubleshooting.
>
> **Acceptance Criteria:** After a visual retrieval query, the INFO log contains an entry with the number of visual results and the stage duration. After a cold-start query, the INFO log contains an entry indicating that the ColQwen2 model was loaded.

## Appendix A. Glossary

| Term | Definition |
|------|-----------|
| BGE-M3 | BAAI General Embedding model (multi-lingual, multi-granularity) used for text embedding in the text retrieval track |
| BGE-reranker | BAAI reranker model used to re-score text retrieval results for improved precision |
| BitsAndBytes | A library for quantizing neural network models to reduce memory usage (e.g., 4-bit quantization) |
| ColBERT | A late-interaction retrieval architecture that scores queries against documents using per-token vector similarity (MaxSim) |
| ColQwen2 | Vision-language model from Vidore that produces cross-modal embeddings for documents and queries |
| Cosine similarity | A measure of similarity between two vectors, computed as the dot product divided by the product of their magnitudes. Range: -1.0 to 1.0 (0.0 to 1.0 for normalized vectors) |
| Docling | A document parsing library that extracts structured content and page images from PDF, PPTX, and DOCX files |
| HNSW | Hierarchical Navigable Small World — an approximate nearest-neighbor graph index used by Weaviate for vector search |
| MaxSim | Maximum Similarity — a scoring function that computes the maximum cosine similarity between each query token vector and all document patch vectors, then sums across query tokens |
| MinIO | S3-compatible object storage used for document and page image storage |
| Named vector | A Weaviate feature that allows a collection to have multiple independently-indexed vector fields, each with its own distance metric and index configuration |
| Presigned URL | A URL that includes query-string authentication parameters, granting temporary access to an object in MinIO/S3 without requiring separate credentials |
| VRAM | Video RAM — GPU memory used for model parameters and inference computation |
| Weaviate | An open-source vector database used for hybrid (keyword + vector) search |

## Appendix D. Open Questions

1. **ColQwen2 text encoder isolation:** Can `colpali_engine.models.ColQwen2` load only the text encoder (without the vision encoder) to save VRAM? If not, the full model must be loaded at query time (~3-4 GB at 4-bit). This directly impacts NFR-903 (VRAM budget). Fallback: load the full model, accept the VRAM cost, and adopt an on-demand load/unload strategy where ColQwen2 is loaded for the visual stage and unloaded immediately after, sacrificing latency for VRAM compliance. *(Related: NFR-903, FR-603)*

2. **VRAM coexistence profiling:** At query time, BGE-M3 and BGE-reranker are already loaded. Adding ColQwen2 means 3 GPU models simultaneously. On RTX 2060 (6 GB), this may require sequential load/unload patterns (embed text -> unload BGE -> load ColQwen2 -> visual search -> unload ColQwen2 -> reload BGE for next query). Implementation should profile actual VRAM usage and select between persistent caching (FR-603) and on-demand loading based on measured headroom. *(Related: NFR-903, FR-603, FR-613)*

3. **Visual result deduplication:** If the same document has many visually-similar pages (e.g., a template-heavy presentation), visual search may return several pages from the same document. Should results be deduplicated per-document (returning only the highest-scoring page per document) or returned raw? The current spec does not require deduplication; this is deferred to implementation-time evaluation. *(Related: FR-301, FR-303)*

4. **Patch-level MaxSim scoring (future consideration):** The `patch_vectors` are stored in Weaviate but not used for retrieval. ColQwen2's native scoring uses MaxSim (sum of max cosine similarities between query token vectors and page patch vectors). Mean-pooling loses fine-grained spatial information. A future iteration could add MaxSim reranking of top-k visual results using the stored patch vectors. This is explicitly out of scope but documented as a known quality improvement path. *(Related: FR-311, out-of-scope section)*
