## 1. Purpose

This specification describes how the OpenTitan `gpio` IP is integrated into the Earl Grey
top-level SoC (`top_earlgrey`). It covers the register interface, interrupt routing, pin
multiplexing constraints, and simulation requirements.

---

## 2. Module Overview

The `gpio` module provides 32 general-purpose bidirectional I/O pins. Each pin can be
independently configured as input or output. The module exposes a TileLink-UL (TL-UL)
device interface for register access and an interrupt line per pin to the interrupt controller
(`rv_plic`).

**Top-level ports (abbreviated):**

| Signal           | Direction | Width | Description                          |
|------------------|-----------|-------|--------------------------------------|
| `clk_i`          | input     | 1     | Primary clock                        |
| `rst_ni`         | input     | 1     | Active-low synchronous reset         |
| `tl_i`           | input     | TL-UL| TileLink upstream port               |
| `tl_o`           | output    | TL-UL| TileLink response port               |
| `cio_gpio_i`     | input     | 32    | GPIO pad inputs                      |
| `cio_gpio_o`     | output    | 32    | GPIO pad outputs                     |
| `cio_gpio_en_o`  | output    | 32    | GPIO output enables (active-high)    |
| `intr_gpio_o`    | output    | 32    | Per-pin interrupt requests           |

---

## 3. Register Interface

The `gpio` register file is generated by `regtool` from `gpio.hjson`. All registers are
32-bit wide and accessible via TileLink-UL. Key registers:

| Register         | Offset | Description                                  |
|------------------|--------|----------------------------------------------|
| `INTR_STATE`     | 0x00   | Interrupt state (W1C)                        |
| `INTR_ENABLE`    | 0x04   | Interrupt enable per pin                     |
| `INTR_TEST`      | 0x08   | Software-driven interrupt test               |
| `DATA_IN`        | 0x10   | Sampled value of `cio_gpio_i`                |
| `DIRECT_OUT`     | 0x14   | Direct write to output register              |
| `MASKED_OUT_LOWER` | 0x18 | Masked write to pins [15:0]                 |
| `MASKED_OUT_UPPER` | 0x1C | Masked write to pins [31:16]                |
| `DIRECT_OE`      | 0x20   | Direct write to output-enable register       |
| `CTRL_EN_INPUT_FILTER` | 0x38 | Enable 16-cycle glitch filter per pin  |

---

## 4. Interrupt Handling

The `gpio` module supports two interrupt detection modes, configurable per pin:

### 4.1 Edge-Detect Mode

When `INTR_CTRL_EN_RISING` or `INTR_CTRL_EN_FALLING` bits are set, the module latches a
rising or falling edge on `cio_gpio_i` and sets the corresponding bit in `INTR_STATE`. The
interrupt remains asserted until software clears it with a write-1-to-clear (W1C) operation.

### 4.2 Level-Sensitive Mode

When `INTR_CTRL_EN_LVLHIGH` or `INTR_CTRL_EN_LVLLOW` bits are set, the interrupt follows the
instantaneous sampled value of `cio_gpio_i`. Level interrupts cannot be cleared while the
condition is active; software must de-assert the external signal first.

---

## 5. SoC Integration (top_earlgrey)

In the top-level SoC, the `gpio` module is instantiated as `u_gpio` and connected:

- **Bus fabric:** TileLink-UL crossbar (`xbar_main`) routes addresses in the range
  `0x4001_0000 – 0x4001_0FFF` to `gpio`.
- **Interrupt controller:** Each bit of `intr_gpio_o[31:0]` fans out to a dedicated input
  of `rv_plic` (PLIC source IDs 24–55).
- **Pinmux:** `cio_gpio_o` and `cio_gpio_en_o` are routed through `pinmux` before reaching
  the physical I/O pads. GPIO pins may be muxed with UART TX/RX, SPI CS, and I2C SDA/SCL
  signals via `pinmux` configuration registers.

---

## 6. Pin Multiplexing Constraints

The following mux conflicts must be resolved in software before enabling GPIO output on
shared pins:

| GPIO Pin | Conflicting IP     | Signal           |
|----------|--------------------|------------------|
| GPIO[0]  | uart_tx            | `cio_uart_tx_o`  |
| GPIO[1]  | uart_rx (loopback) | `cio_uart_rx_i`  |
| GPIO[8]  | spi_fwmode         | `cio_spi_cs_o`   |
| GPIO[9]  | spi_fwmode         | `cio_spi_sck_o`  |

Software must write `PINMUX_MIO_OUTSEL` and `PINMUX_MIO_PERIPH_INSEL` to resolve these
conflicts. Driving conflicting output enables simultaneously is undefined behavior.

---

## 7. Simulation

Functional simulation is performed with **Verilator** (v5.018). The cocotb testbench
`sim_gpio.py` exercises:

- Register read/write round-trips via TileLink BFM
- Rising-edge interrupt on GPIO[3]
- Level-high interrupt latching and W1C clear
- Glitch filter timing (requires 16 consecutive same-level samples)

Run simulation with:

```bash
make -C hw/ip/gpio/dv SIM=verilator
```
